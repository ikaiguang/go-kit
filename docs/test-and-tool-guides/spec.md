# Test And Tool Guides

## 背景

用户要求检查所有包含 Go 文件的目录，确认已有测试是否可用；没有测试时补充对应测试；运行测试验证功能正确性；如发现功能问题则优化、纠正或修复。同时需要为每个工具补充使用指导文档。

本仓库遵循 `AGENTS.md` 和 `.agents/skills/my-project/` 的项目约束：先读取目标模块和相邻实现，保持最小改动，使用 `gofmt`，测试以 `go test` 为主。

## 目标

- 枚举并检查仓库中包含 Go 文件的目录。
- 检查已有 `*_test.go` 是否覆盖核心功能并能通过测试。
- 对缺少必要测试的工具目录补充聚焦的单元测试。
- 运行测试验证功能正确性，发现功能、安全、性能、死循环、非必要 `panic` 等问题时做最小必要修复。
- 为每个工具补充使用指导文档，说明用途、典型用法、注意事项和测试方式。

## 非目标

- 不重写整体架构，不做无关重构。
- 不手动修改 Proto 生成文件。
- 不引入新的外部服务依赖或系统级依赖。
- 不发布、部署或改写 Git 历史。

## 影响范围

- 预计重点影响包含工具代码的 Go 目录及其相邻测试文件。
- 预计新增或修改 `*_test.go`。
- 预计新增或修改工具使用指导文档，具体位置会优先沿用仓库已有文档结构；若没有合适位置，放在 `docs/` 下按工具分组。
- 可能读取 `.agents/skills/my-project/references/` 中的项目上下文、编码规则和命令说明。

## 方案

1. 先读取项目上下文、编码规则、命令说明，以及仓库中 Go 包分布。
2. 使用 `go list`、`rg --files` 等方式枚举包含 Go 文件的目录，区分工具包、服务示例、生成代码和测试数据。
3. 对每个目标目录检查是否已有测试，运行对应测试确认现状。
4. 对缺少测试或测试不足的工具代码，基于相邻实现补充最小有效测试。
5. 若测试或代码检查暴露功能问题、安全问题、性能问题、死循环风险、非必要 `panic` 等问题，先定位原因并做最小修复，再补充或调整测试验证；如修复会扩大影响范围，先更新本规格再继续。
6. 为每个工具编写使用指导文档，优先包含：用途、适用场景、基础用法、示例、测试/验证命令、注意事项。
7. 运行必要的 `gofmt` 和 `go test` 验证。

## 验收标准

- 包含 Go 文件的目标目录均经过测试现状检查。
- 工具目录有对应测试，且核心功能可被本地测试验证。
- 已检查并处理测试或阅读代码时发现的安全、性能、死循环和非必要 `panic` 风险。
- 工具使用指导文档可帮助开发者理解如何使用和验证这些工具。
- 相关 `go test` 命令通过，或明确记录无法通过的原因和剩余风险。
- 所有代码改动已 `gofmt`。

## 风险与回滚

- 风险：仓库目录较多，逐个补全测试可能扩大任务范围。规避方式：优先聚焦“工具”目录，对生成代码、第三方样式代码、纯示例入口按实际价值判断是否补测。
- 风险：部分测试可能依赖外部服务。规避方式：优先编写纯单元测试；遇到外部依赖时用 mock、fake 或跳过集成验证并记录原因。
- 风险：修复功能、安全、性能、死循环或非必要 `panic` 问题可能影响公共 API 或既有行为。规避方式：先做最小修复并补测试；如需要改公共接口或跨模块依赖，先更新规格并等待确认。
- 回滚：本次改动均限定在当前 workspace，可通过版本控制查看并回退相关测试、源码和文档文件。
